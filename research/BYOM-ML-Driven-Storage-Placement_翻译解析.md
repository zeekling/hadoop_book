## 一、摘要翻译

存储系统在仓库级计算机的总拥有成本（TCO）中占很大比例，因此对整体系统效率有重大影响。基于机器学习的方法在解决存储系统效率的关键问题（如数据放置）方面展现出巨大潜力。然而，这类方法的实际部署案例极少。

通过在 Google 真实超大规模数据中心研究此问题，我们识别出导致这一部署缺失的多项挑战。具体而言，先前的工作假设存储层内部驻留一个**单体模型**(monolithic model)，这在工作负载频繁变化的真实部署中是不现实的假设。

为解决此问题，我们引入一种**跨层方法**，让工作负载"自带模型"（Bring Your Own Model, BYOM）。该策略将 ML 移出存储系统，转而允许每个工作负载在应用层训练自己的轻量级模型，捕获该工作负载的特定特征。这些小型、可解释的模型生成预测，指导存储层中协同设计的调度启发式算法，从而适应多样化的在线环境。

我们在 Google 的生产分布式计算框架中构建了概念验证系统。在测试部署和大规模仿真研究中，使用生产环境的 trace 进行评估，结果表明，相比最先进的基线方法，TCO 节省最高可达 **3.47 倍**。

## 二、研究背景与动机

### 2.1 存储分层问题

仓库级数据中心中，存储系统是 TCO 的主要组成部分。即使 1% 的改进也代表巨大的成本节约。**数据放置**（决定文件存放在 HDD 还是 SSD）即存储分层（storage tiering），是一个关键决策。

当前主流方法分为两类：

| 方法 | 描述 | 优缺点 |
|------|------|--------|
| **启发式方法** (Heuristics) | 贪婪分配数据到 SSD 直至容量用尽，超出的放到 HDD | 快速可解释，但 SSD 容量有限时表现次优 |
| **ML 方法** | 利用真实工作负载信息训练模型做决策 | 仿真效果好，但极少实际部署 |

### 2.2 现有 ML 方法的部署障碍

作者在 Google 的真实生产环境中分析发现，现有 ML 方法难以部署的主要原因：

1. **单点故障风险** — 一个单体 ML 模型驻留在存储层，成为单点故障
2. **关注点分离被破坏** — 工作负载和应用频繁变化（文件改名、新应用上线等），但存储层更新周期远慢于应用层。模型在存储层意味着应用变化触发模型重训练
3. **输入特征局限** — 最具预测性的特征往往是工作负载特定的，难以可靠传递到存储层
4. **运行时开销** — 如 Transformer 模型推理延迟约 99ms，对在线决策不可接受
5. **适应性问题** — 静态模型无法适应动态变化的工作负载模式

### 2.3 核心洞察

数据中心的核心理念是通过**抽象层**（应用层、存储层、硬件层）分离关注点。每一层独立演化和优化。现有 ML 方案试图用一个端到端的存储层内置模型来处理所有问题，**打破了这种抽象隔离**。

> **BYOM 的核心思想**: 不把 ML 模型嵌入存储层，而是让每个工作负载在应用层"自带模型"，存储层只负责根据各应用提供的"提示"(hint) 做调度决策。

## 三、问题定义与基线方法

### 3.1 问题形式化

基本数据放置单元：**shuffle job**（数据洗牌作业）。

每个 job 有四个属性：`(start_time, lifetime, job_size, cost)`。

优化目标：在有限的 SSD 容量下，最大化将 job 放在 SSD 上的成本节约。

**ILP 形式化**:

```
max  Σ_i x_i (c_HDD_i - c_SSD_i)
s.t. x_i ∈ {0,1}, ∀i
     Σ p_i(t) ≤ M, ∀t   (SSD 容量约束)
```

### 3.2 TCO 与 TCIO 指标

| 指标 | 全称 | 说明 |
|------|------|------|
| **TCO** | Total Cost of Ownership | 总拥有成本：包含存储、网络、服务器、SSD 磨损等全部费用 |
| **TCIO** | Total Cost of I/O | 仅 I/O 成本，不含 SSD 磨损和网络成本，辅助理解纯 I/O 视角的优化 |

TCO 计算公式：
```
TCO_DEV = cost_byte + cost_network + cost_server + cost_specific
```

### 3.3 基线方法

| 方法 | 类型 | 描述 |
|------|------|------|
| **Oracle** | 理论最优 (ILP) | 全知视角下的最优解，有先见之明，不可部署 |
| **FirstFit** | 简单启发式 | 按作业到达顺序贪心放置到 SSD，先到先得 |
| **Heuristic** | 高级启发式 | 基于作业类别的自适应 admission 策略，动态学习 (Yang et al., 2022) |
| **ML Baseline** | 机器学习 (Lifetime) | 预测文件生存期，短于 TTL 的放 SSD (Zhou & Maas, 2021) |
| **Adaptive Hash** | 无 ML 的 BYOM | 用 hash 代替 ML 模型，无学习能力 |
| **Adaptive Ranking** | **本文方法** | BYOM + 梯度提升树分类模型 |

## 四、BYOM 混合学习方法（核心贡献）

### 4.1 设计哲学

关键洞察：数据中心是**高度动态**的环境，最优放置决策不仅取决于工作负载特征，还取决于**外部环境因素**（如当前可用 SSD 容量、其他工作负载的存在等）。

传统端到端学习（如模仿学习）需要模型同时学习**外部环境**和**工作负载**，因而不切实际。

**BYOM 方案**：
- 模型 **只负责预测工作负载特定特征的代理指标**
- 存储层的调度启发式算法负责将预测转化为**适应当前环境的决策**
- 这样模型和存储层解耦，各自独立演进

### 4.2 "重要性"代理指标

作者设计了一个称为 **"importance"（重要性）** 的代理指标。

**原理**：
- 用 Oracle 在不同 SSD 容量下计算最优放置决策
- 观察发现：高重要性 job = 更高 I/O 密度 + 正 TCO 节约
- 因此用 **(TCO savings, I/O density)** 二元组作为分类标签

**标签设计**：
- 负 TCO 节约的 job → 类别 0（最低重要性）
- 正 TCO 节约的 job → 按 I/O 密度分位数分为 N-1 个类别

### 4.3 模型设计

- **模型类型**: 梯度提升树（Gradient Boosted Trees）
- **框架**: Yggdrasil Decision Forests (YDF)
- **类别数**: 15 类（1 类负 TCO + 14 类按 I/O 密度分位数）
- **树木参数**: 最多 300 棵树，最大深度 6
- **训练粒度**: 按集群（cluster）训练，每个集群一个模型

**为什么不选其他模型**:

| 模型 | 问题 |
|------|------|
| Transformer / 神经网络 | 推理延迟高（~99ms），不可解释 |
| 查找表 | 精度不够 |
| 梯度提升树 | 平衡了轻量高效和精度（~4ms/推理） |

**特征组**（四组）：

| 组 | 特征 | 说明 |
|----|------|------|
| A | 历史系统指标 | 过去 TCIO、job 生存期、大小等 |
| B | 执行元数据 | 执行相关名称、路径、目标等 |
| C | 分配资源 | 调度器分配的 CPU/内存等资源 |
| T | 时间戳 | 星期几、小时等周期性特征 |

### 4.4 自适应类别选择算法

核心算法（Algorithm 1）：

1. 每个 job 到达时，模型预测其**重要性类别**
2. 存储层维护一个**滑动类别准入阈值**（Admission Category Threshold, ACT）
3. 引入 **spillover TCIO 百分比** 指标衡量 SSD 压力：
   - spillover 大 → SSD 接近满 → 提高阈值（减少准入类别）
   - spillover 小 → SSD 有余量 → 降低阈值（增加准入类别）
4. 使用**容错范围**和**固定决策间隔**避免阈值剧烈波动

**关键指标 — Spillover TCIO 百分比**:
```
P_spillover = Σ spillover_TCIO / Σ job_TCIO_HDD
```
衡量因 SSD 容量不足而被迫放到 HDD 的 job 所占的 TCIO 比例。

### 4.5 系统集成架构


## 五、实验评估

### 5.1 实验设置

| 维度 | 细节 |
|------|------|
| 数据来源 | Google 生产环境 trace（连续两周） |
| 训练/测试 | 各一周数据 |
| 集群规模 | 数千台机器 |
| 评估集群 | 10 个高 TCO 集群 |
| 工作负载 | 视频处理、ML、数据库查询等 |
| 端到端原型 | 320 台 worker、16 条 pipeline、1024 个 shuffle job、峰值 3.6 TiB |

### 5.2 端到端系统集成结果（RQ1）

SSD 配额为峰值 1.0% 时：
- **Adaptive Ranking（本文方法）**: 1.14% TCO 节约
- **FirstFit**: 0.26% TCO 节约
- **提升**: **4.38 倍**

SSD 配额为峰值 20% 时：
- **Adaptive Ranking**: 2.48% TCO 节约
- **FirstFit**: 1.40% TCO 节约
- **提升**: **1.77 倍**

### 5.3 大规模仿真结果（RQ2, RQ3）

**主要结果**（SSD 配额 1.0%，10 个集群平均）:

| 方法 | TCO 节约 (%) | 与最优基线对比 |
|------|-------------|---------------|
| **Adaptive Ranking (本文)** | **~2.5-4.5** | **3.47×（最大） / 2.59×（平均）** |
| Adaptive Hash (无 ML) | ~0.5-1.5 | — |
| ML Baseline (Lifetime) | ~0.5-1.0 | — |
| FirstFit | ~0.2-0.8 | — |
| Heuristic | ~0.8-1.8 | — |
| Oracle TCO (理论最优) | ~5-12 | 上限参考 |

**不同 SSD 配额下的表现**: Adaptive Ranking 在各种 SSD 配额下均优于所有基线，尤其在**SSD 有限时差距最大**（实用中最常见场景）。

### 5.4 泛化性（RQ4）

- **跨集群泛化**: 在 C0 训练的模型在 C1/C2 表现良好（普通集群），C3（特殊工作负载集群）表现略差
- **新用户/新 pipeline**: 排除目标用户或 pipeline 训练的模型与包含的模型性能相近，说明方法对未见过的数据有良好泛化性

### 5.5 模型分析（RQ5）

- **推理延迟**: ~4ms/job（远低于 Transformer 的 99ms）
- **Top-1 准确率**: 0.36（15 分类）
- **训练规模相关性**: 准确率与训练数据量弱相关，扩大数据未必提升效果
- **特征重要性**: 历史系统指标 (A 组) 对预测 I/O 密度排名影响最大，时间戳和元数据次之

### 5.6 真实类别 vs 预测类别对比

使用真实类别（ground truth category）替代模型预测时，效果接近但略好，说明**模型自身不是瓶颈**，算法的设计才是关键。

## 六、结论

1. **BYOM 是一种实用的 ML+Systems 设计范式**: 将 ML 模型部署在应用层，存储层仅通过简单接口消费模型输出的"提示"，保持了数据中心的层次化抽象。

2. **跨层设计 = 解耦 + 适应性**: 模型预测工作负载特定属性，调度算法根据实时环境调整决策，两者互为补充。

3. **真实系统验证可行**: 在 Google 生产环境的概念验证表明，该方法可以在不修改用户代码的情况下集成，并带来显著 TCO 节约（3.47×）。

4. **梯度提升树是实用选择**: 相比 Transformer，推理速度快 25 倍，同时保持了足够的预测能力。

## 七、个人总结与学习要点

### 对系统研究者的启示

1. **关注"为什么 ML 没有实际部署"比"ML 能多好"更重要** — 本文最大的贡献不在于提出新模型，而在于深入分析了 ML 在真实系统中部署失败的原因，并设计了实用化的解决方案。

2. **跨层设计是 ML+Systems 的关键模式** — 不把 ML 塞进已有的系统层，而是在各层之间设计清晰的接口，让 ML 作为独立消费者提供服务。

3. **代理指标（proxy）的力量** — 不直接预测"放 SSD 还是 HDD"（这是一个依赖于环境的问题），而是预测工作负载本身的固有特征（I/O 密度），让调度算法根据环境做出决策。

### 对工程实践的启示

4. **可解释性 + 低延迟 = 可部署性** — 在系统领域，模型不仅要准，还要快、可解释、可调试。梯度提升树优于黑盒神经网络。

5. **反馈闭环设计** — spillover TCIO 作为在线反馈指标，形成闭环控制系统，使系统能自动适应环境变化。

6. **增量验证策略** — 先做 headroom analysis（ILP Oracle 算理论上限），再设计实用方法，确保方向正确。
