## 摘要翻译

现代分布式文件系统（DFS）依赖元数据服务器集群来管理大规模文件并实现可扩展性。然而，层次化的命名空间结构和动态的用户负载给高效的元数据分区和负载均衡带来了严峻挑战。

现有方法主要聚焦于识别和重新分布热点元数据来解决不均衡问题。虽然这些负载均衡策略有潜在收益，但它们往往会降低元数据局部性，最终无法改善端到端作业完成时间——这是用户优先关注的关键指标。

尽管最近的研究表明基于学习的方法在预测热点方面是有效的，但它们在提升元数据性能方面效果不佳。我们重新审视了元数据负载均衡策略，提出了基于学习的元数据负载均衡框架 **Origami**，其核心是**最小化端到端作业完成时间**而非均衡负载。

Origami 首先分解元数据操作的开销，评估迁移决策对用户请求的影响，从而在已知未来请求时计算迁移决策对作业完成时间的收益。随后，Origami 提出了 **Meta-OPT 算法**来确定近最优的迁移决策。最后，我们实现了 **OrigamiFS**，在其上收集统计数据来训练和验证能够预测迁移收益的 ML 模型。

实验评估表明，与最先进方法相比，Origami 在三个真实工作负载上将聚合元数据吞吐量提升了 **1.12–2.51 倍**，端到端吞吐量提升了 **1.11–2.02 倍**。

## 1. 研究动机

### 1.1 元数据已成为 DFS 主要瓶颈

- 现代数据中心文件规模达数千亿，平均文件大小从 GB 降至 MB
- 超过 **90%** 的 I/O 请求小于 1MB，元数据操作占比持续上升
- 单台 MDS 已无法满足处理速度和存储容量需求

### 1.2 关键发现：均匀分布反而有害

作者通过 CephFS 集群实验（5-MDS + 50 客户端）发现：
- 5 个 MDS（均匀哈希分布）的单个 MDS 吞吐量**显著低于**单 MDS 配置
- 聚合吞吐量仅提升约 **1.4 倍**（对比理想 5 倍）
- 作业完成时间仅减少 **57%**

**根因**：路径解析需要多次 RPC 遍历，元数据迁移破坏命名空间局部性，引入额外开销。

### 1.3 现有方法的局限性

| 方法 | 代表系统 | 问题 |
|------|---------|------|
| **粗粒度分区** | CephFS 动态子树 | 难扩展，动态负载下易产生热点 |
| **细粒度分区** | Tectonic/InfiniFS 哈希 | 破坏局部性，延迟随目录深度线性增加 |
| **ML 热点预测** | 近期 ML 方法 | 忽略层次结构，动态负载下预测效率低 |

### 1.4 三个核心挑战

1. **评估适当的指标**：QPS 或 inode 数不能反映局部性破坏对用户任务的负面影响
2. **寻找有效的迁移决策**：命名空间深度超 10 层，父子节点迁移策略相互干扰
3. **收集统计数据与验证模型**：现有 DFS 非 ML 原生，难以收集训练数据

## 2. Origami 方法与技术创新

### 2.1 核心理念

> **均匀分布元数据负载不应是主要优化目标。**  
> 应聚焦于优化 MDS 集群效率、最小化端到端作业完成时间（JCT）。

### 2.2 请求完成时间（RCT）度量

将元数据操作开销分解：

```
RCT = T_meta + m·RTT + ΣQ_i
```

其中：
- `T_meta` = 元数据处理开销（固定基线 + 可变额外开销）
- `m` = 涉及的元数据分区数
- `RTT` = 网络请求访问时间
- `Q_i` = 各分区排队时间

元数据操作分为三类，迁移后额外开销各不相同：

| 类型 | 迁移后额外开销 |
|------|--------------|
| **目录列表（lsdir）** | `i·RTT`（子文件/目录迁移到 i 个其他 MDS） |
| **命名空间变更（ns-m）** | `T_coor`（分布式事务协调开销） |
| **其他操作** | 无额外开销 |

### 2.3 Meta-OPT 算法

受经典 Bélády 算法启发，设计 Meta-OPT 算法：

**核心思想**：
1. 给定已知的未来元数据操作序列 N 和当前 MDS 状态，迭代寻找能最大化收益（减少总体完成时间）的迁移决策
2. 每次迭代贪心地选择收益最大的子树进行迁移
3. 设置阈值 Δ 防止迁移导致新的不均衡
4. 直到所有迁移收益低于阈值停止

**理论保证**（Theorem 1）：即使贪心策略非最优，与最优解的差距也小于 Δ。

### 2.4 OrigamiFS 架构

使用 **Go 语言**实现的轻量级分布式元数据服务：

**组件**：
- **Metadata Cluster**：MDS 0~n 组成，每个 MDS 使用 PebblesDB（KV存储）存储 inode
- **Data Collector**：导出运行时命名空间状态，收集 ML 训练特征数据
- **Migrator**：执行外部迁移决策（支持 Meta-OPT 和 ML 模型 pipeline 输入）
- **Metadata Balancer**（MDS-0）：贪心选择收益最高的子树迁移
- **OrigamiFS SDK**：客户端库，含**近根元数据缓存**（深度 < 阈值，缓存 < 1% 的元数据）

### 2.5 工作流

```
① 回放真实工作负载 → ② Data Collector 收集统计 → 
③ Meta-OPT 计算迁移收益（作为训练标签）→ ④ 执行决策并继续收集 → 离线模型训练
```

### 2.6 ML 模型

- **模型选择**：比较了 LightGBM、GBDT、MLP（4隐藏层），最终选择 **LightGBM**
  - 三者预测准确率相近，迁移决策高度相似
  - LightGBM 预测开销最小（400 轮 boosting，32 叶子）
- **训练特征**（7 个）：

| 类别 | 特征 | 归一化方式 | Gini 重要性 |
|------|------|-----------|-----------|
| 命名空间结构 | depth（目录深度） | max 归一化 | 7 |
| | # sub-files（子文件数） | — | 1 |
| | # sub-dirs（子目录数） | — | 4 |
| 元数据历史 | # read（读次数） | 除以上周期总访问数 | 6 |
| | # write（写次数） | 同上 | 2 |
| 派生特征 | read-write ratio（读写比） | 原始值 | 6 |
| | dir-file ratio（目录文件比） | 原始值 | 2 |

- **模型验证**：在 OrigamiFS 上在线验证模型，直接评估集群性能优化效果，而非仅依赖准确率指标

## 3. 实验评估

### 3.1 实验设置

| 项目 | 配置 |
|------|------|
| **集群** | 10 节点 Kubernetes，5-MDS + 5 客户端 |
| **节点规格** | 8 CPU + 64GB RAM + 2TB NVMe SSD |
| **工作负载** | Trace-RW（读写混合编译任务）、Trace-RO（只读 Web 访问，深度偏斜）、Trace-WI（写密集型云文件系统） |

### 3.2 总体性能

| 指标 | C-Hash | F-Hash | ML-tree | **Origami** | 单 MDS |
|------|--------|--------|---------|------------|--------|
| 聚合吞吐量（高负载） | 2.23× | 1.54× | 1.89× | **3.86×** | 1× |
| 平均延迟（单线程） | +43.9% | +89.1% | +29.3% | **+24.2%** | 基准 |

**结论**：Origami 在高负载时最大化吞吐量，低负载时最小化性能退化。

### 3.3 缓存分析

| 指标 | C-Hash | F-Hash | ML-tree | **Origami** |
|------|--------|--------|---------|------------|
| 无缓存吞吐量 | 32.8k | 22.5k | 26.7k | **39.3k** |
| 有缓存吞吐量 | 46.0k | 30.0k | 38.6k | **78.9k** |
| 缓存后每请求额外 RPC | 0.54 | 1.27 | 0.17 | **0.035** |

**发现**：Origami 倾向于迁移两类子树——高负载的近根节点（单次迁移显著改善均衡）和写密集的远端节点（少量元数据操作但收益大）。

### 3.4 可扩展性

- **3-MDS 时**：接近线性扩展（单 MDS 的 2.7 倍）
- **5-MDS 时**：增速放缓但仍为最优

### 3.5 真实工作负载结果

| 场景 | 元数据吞吐量（vs 次优基线） | 端到端吞吐量 |
|------|---------------------------|------------|
| Trace-RW（读写混合） | +73.3% | +37% |
| Trace-RO（只读） | +54.3% | +26% |
| Trace-WI（写密集） | +12.5% | +11% |

## 4. 学习要点

1. **均匀 ≠ 最优**：在层次化命名空间中，均匀分布元数据带来的 RPC 开销可能超过均衡收益
2. **以 JCT 为目标**：负载均衡应直接优化用户感知的端到端作业完成时间，而非中间指标（QPS、负载均衡度）
3. **ML 需要框架支撑**：有效的 ML 元数据管理不仅需要好的算法，还需要 ML 原生的系统设计（特征收集、在线验证）
4. **贪心策略可行**：Meta-OPT 的贪心算法有理论保证（误差 ≤ Δ），实际效果优于全局最优搜索
5. **缓存与迁移协同**：近根缓存大幅提升迁移收益，Origami 的迁移模式与缓存策略天然互补
