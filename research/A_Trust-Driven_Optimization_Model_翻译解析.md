## 摘要

授权是 Hadoop 环境中的一项基础服务。它在确保数据和协议安全的同时，仅允许授权用户以受控方式执行数据存储、处理和文件传输等操作。然而，由于 Hadoop 生态系统的漏洞以及潜在的用户异常行为，维持有效的授权仍然充满挑战。目前已开发了多种基于不同假设的授权协议，各自通过不同的技术手段确保可靠高效的访问控制。这些技术各有优缺点，授权可通过集成额外的安全机制进一步增强。

本文提出了一种基于信任的授权模型，帮助 NameNode 在访问控制决策中发挥作用，特别是在处理不确定或不可信的用户时。该模型解决两个关键问题：(1) 认证用户是否应被授权执行请求的操作？(2) 请求应如何执行？

我们的方法利用机器学习以及监控系统和入侵检测机制等现有安全工具，识别入侵或未授权操作。通过持续分析传入请求并监控 Hadoop 环境，该方法在保持操作效率的同时增强了安全性。此外，我们集成了**象群优化（Elephant Herding Optimization, EHO）** 算法，实现分布式自适应的访问控制机制，优化用户满意度、最小化安全风险并减少资源消耗。

基于角色访问控制（RBAC）和属性访问控制（ABAC）的通用授权框架，使用不同性能指标对提出的模型进行了评估。实验结果证明了该方法在检测未授权访问方面的有效性。

## 各章节翻译与详解

### 1. 引言

#### 翻译

Hadoop 已成为大数据分析领域最强大、应用最广泛的框架之一。其核心安全挑战之一是**授权（Authorization）**——决定已认证用户能否执行某个操作。传统授权机制（如 Kerberos 结合 ACL）存在静态性强、难以应对未知威胁等问题。

本文提出了 **ITM-RAH（Intelligent Trust Model-based Reliable Authorization in Hadoop，基于智能信任模型的 Hadoop 可靠授权）** 框架，核心贡献包括：

1. 对现有授权协议的系统分类：**基于访问控制策略的协议** 和 **基于异常行为检测的协议**
2. 提出信任驱动的优化模型，将用户分为**可信（Trusted）、不确定（Uncertain）、不可信（Untrusted）** 三类
3. 开发了新的用于入侵检测的数据集
4. 基于**象群优化（EHO）** 的多目标优化方法
5. 基于机器学习的入侵检测系统的全面评估

#### 解析

引言清晰定义了问题域：Hadoop 授权面临的核心挑战是静态 RBAC/ACL 无法适应动态安全威胁。作者选择将**信任管理 + 机器学习 + 群智能优化**三者融合，形成混合式安全框架。其提出的二分类法（策略 vs. 行为检测）是对现有文献的有价值归纳。

### 2. 背景与相关工作

#### 翻译

##### 2.1 授权协议面临的核心挑战

本节详细分析了 Hadoop 环境中授权遇到的主要问题：

- **Kerberos 的局限性**：Kerberos V5 是 Hadoop 的标准认证协议，但它仅解决"你是谁"的问题，不解决"你应该能做什么"的问题。
- **ACL（访问控制列表）的僵化**：HDFS 的 POSIX 风格 ACL 缺乏细粒度控制。
- **Apache Ranger 和 Apache Sentry**：提供了更细粒度的授权，但仍然基于静态策略。
- **内部攻击威胁**：合法用户可能滥用权限——这是最难防御的攻击向量之一。

##### 2.2 现有授权协议分类

**第一类：基于访问控制策略的协议**
- **RBAC（基于角色的访问控制）**：按角色分配权限，简单但粗粒度。
- **ABAC（基于属性的访问控制）**：基于用户/资源/环境的属性动态决策，更灵活。
- **混合模型**：如 OT-RBAC、HeABAC 等。
- **区块链增强**：用不可篡改的账本存储授权决策记录。

**第二类：基于异常行为检测的协议**
- **入侵检测系统（IDS）**：监控网络流量和系统日志。
- **用户行为分析（UBA）**：建立用户行为基线，检测异常偏离。
- **机器学习方法**：用分类器识别恶意活动。

#### 解析

背景部分对 Hadoop 生态的安全工具有全面梳理。Kerberos 被定位为"认证"而非"授权"工具是准确的。Apache Ranger 和 Sentry 提供了基于策略的授权，但缺乏动态信任评估能力——这正是本文的关键切入点。

### 3. 问题陈述与贡献

#### 核心问题

1. 现有授权机制无法动态评估用户的信任度。
2. 单靠静态策略无法应对高级持续性威胁（APT）和内部攻击。
3. 缺乏在安全、用户体验、资源消耗之间的多目标优化。

#### 五个核心贡献

| 编号 | 贡献 | 说明 |
|------|------|------|
| C1 | 现有 Hadoop 授权协议的系统分类 | 按策略驱动/行为驱动两大类别组织文献 |
| C2 | ITM-RAH 信任驱动授权模型 | 集成 RBAC + ABAC + IDS + 监控系统 |
| C3 | 新的入侵检测数据集定义 | 18 个特征（见第 4 节），但未实际构建 |
| C4 | 基于 EHO 的多目标优化 | 同时优化安全风险、资源消耗、用户满意度 |
| C5 | ML-IDS 的全面评估 | 10 个子集、5 种分类器、7 个评估指标 |

### 4. 提出的模型 ITM-RAH

这是论文的核心技术部分，分为四个子模块。

#### 4.1 信任与不信任的度量

**信任值（TV, Trust Value）**：

```
TVe = (H × α) + (1 - NDA / Max) × β + (1 - ASR) × γ
```

其中：
| 符号 | 含义 |
|------|------|
| TVe | 当前会话 e 中用户的信任值 |
| H | 诚实度度量（0 = 不诚实, 1 = 诚实） |
| NDA | 该用户被 IDS 检测到的攻击次数 |
| Max | 历史记录中的最大攻击次数 |
| ASR | 攻击成功率 = 成功攻击次数 / 总攻击尝试 |
| α, β, γ | 权重系数（α+β+γ=1） |

**不信任值（UTV, Untrust Value）**：

```
UTVe = (SF × λ) + (ALF × μ) + (ABF × ν)
```

其中 SF = 安全事件频率因子，ALF = 异常登录频率，ABF = 异常行为频率。

##### 信任交互规则

信任值变化基于安全日志检测到的事件：

| 场景 | 信任值变化 |
|------|-----------|
| IDS 检测到 Hadoop 漏洞利用尝试 | 显著降低 |
| 多次登录失败（暴力破解） | 降低 |
| 资源消耗异常或可疑流量 | 中度降低 |
| 检测到恶意软件执行 | 清零 → 立即封号 |
| 持续合法行为 | 逐步恢复（信任恢复机制） |

#### 4.2 信任值的区块链存储

信任数据以区块链不可篡改格式存储：

```
{ID_user, TVe, Status_user, Time_of_storage}
```

- 仅 Hadoop 管理员有权限更新
- 区块链防止未授权修改，确保透明性和可问责性

#### 4.3 机器学习模型安全（对抗性机器学习防御）

针对**对抗性机器学习（AML）攻击**，提出两个防御策略：

**策略一：多分类器集成**
- 不使用单一分类器，而是用多个分类器的集成
- 攻击者难以预测哪个分类器将被用来评估特定请求
- 增加了攻击者的不确定性

**策略二：新数据集定义**

论文提出了一个新的数据集 schema（Table 3），包含 18 个特征：

| # | 特征 | 可能值 |
|---|------|--------|
| 1 | 年龄 | ≥18 或 <18 |
| 2 | 性别 | 男/女 |
| 3 | 心理或精神障碍 | 是/否 |
| 4 | 有工作 | 是/否 |
| 5 | 有家庭 | 是/否 |
| 6 | 生活质量 | 好/差 |
| 7 | 自我动机（复仇、宗教、商业利益等） | 是/否 |
| 8 | 与管理员/分析师/开发者的友谊 | 是/否 |
| 9 | 与黑客有友谊或是黑客 | 是/否 |
| 10 | 社会声誉 | 诚实/不诚实 |
| 11 | 是否为使用 Hadoop 的已认证用户 | 是/否 |
| 12 | 连续攻击次数 | ≥0 |
| 13 | 最近一次攻击日期 | 是否是活跃攻击者 |
| 14 | 危险攻击次数 | ≥0 |
| 15 | 使用的操作系统 | Linux/Mac/Android 等 |
| 16 | 编程水平 | 高级/良好 |
| 17 | 使用的硬件 | 高级/良好 |
| 18 | 位置信息 | 国家、经纬度等 |

> ⚠️ 作者明确指出该数据集的准备是**未来工作**，当前未实际构建。

#### 4.4 基于 EHO 的多目标优化

**象群优化算法（Elephant Herding Optimization, EHO）** 受大象群体行为启发：

- **家族（Clan）更新操作**：解决方案在群体间交换知识，协同进化
- **分离（Separation）操作**：保持解的多样性，避免过早收敛

##### 解决方案编码

每个解决方案（大象个体）代表一个授权决策，含三个优化目标：

| 指标 | 范围 | 含义 |
|------|------|------|
| **安全风险 (SR)** | [0,1] | 策略相关安全风险，含误报率、影响评估 |
| **资源消耗成本 (RCC)** | [0,1] | 能耗、RAM 使用、CPU 负载 |
| **用户满意度 (US)** | [0,1] | 响应时间、正向响应概率 |

##### 适应度函数

```
Fitness = w1 × (1-SR) + w2 × (1-RCC) + w3 × US
```

实验中权重设定：w1=0.5, w2=0.3, w3=0.2（安全优先级最高）

##### 六种授权策略

| 策略 | 动作 | 适用场景 |
|------|------|---------|
| Policy 1 | 直接执行请求 | 用户完全可信 |
| Policy 2 | 在隔离环境（蜜罐/沙箱）中执行 | 不确定用户 |
| Policy 3 | 隔离环境中执行有限任务 | 低风险不确定用户 |
| Policy 4 | 短期拒绝服务请求 | 可疑行为 |
| Policy 5 | 长期拒绝服务请求 | 高危行为 |
| Policy 6 | 从数据集中删除该用户 | 不可信用户 |

##### 用户分类算法（Algorithm 2）

```
输入：TVe, UTVe, He（当前会话）
对每个未阻塞用户的服务请求：
  1. 计算 TVe, UTVe
  2. 若 H=0 → 不可信（Untrusted）
  3. 若 H=1:
     - TVe > UTVe → 可信（Trusted）
     - 否则 → 不确定（Uncertain）
  4. 若 H≠0,1:
     - TVe ≥ UTVe → 不确定（Uncertain）
     - 否则 → 不可信（Untrusted）
```

### 5. 性能评估

#### 5.1 基于 ML 的入侵检测系统评估

##### 实验设置

| 项目 | 配置 |
|------|------|
| 操作系统 | Ubuntu 22.04 LTS |
| CPU | Intel Core i5-3337U @ 1.80GHz |
| RAM | 6 GB |
| 实现语言 | Python 3 |
| 数据集 | CIC-IDS-2017（6 子集）, CIC-DDoS-2019（4 子集） |
| 数据划分 | 70% 训练 / 30% 测试 |
| 数据预处理 | 缺失值检测、字母→数字转换、归一化、特征选择 |

##### 五种分类器

| 分类器 | 特点 |
|--------|------|
| **Random Forest (RF)** | 集成学习，多决策树组合，精度高但训练慢 |
| **Decision Tree (DT)** | 单一决策树，可解释性强 |
| **K-Nearest Neighbors (KNN)** | 基于距离的惰性学习，无显式训练 |
| **XGBoost** | 梯度提升优化树，高效高精度 |
| **LightGBM** | 轻量梯度提升，训练最快之一 |

##### 实验结果汇总

**CIC-IDS-2017 数据集（6 子集）：**

| 子集 | 攻击类型 | 最佳分类器 | 最佳准确率 |
|------|---------|-----------|-----------|
| DS1-1 | DDoS | RF / XGBoost | 99.96% |
| DS1-2 | PortScan | RF / XGBoost | 99.99% |
| DS1-3 | Bot | RF | 98.60% |
| DS1-4 | Web Attacks (XSS, Brute Force, SQL Injection) | RF | 99.20% |
| DS1-5 | FTP-Patator, SSH-Patator | LightGBM | 96.61% |
| DS1-6 | DoS slowloris, slowhttptest | KNN | 92.49% |

**CIC-DDoS-2019 数据集（4 子集）：**

| 子集 | 最佳分类器 | 最佳准确率 |
|------|-----------|-----------|
| DS2-1 (LDAP) | RF / XGBoost | 100% |
| DS2-2 (Portmap) | RF / LightGBM | 99.99% |
| DS2-3 (Syn) | XGBoost / LightGBM | 98.17% |
| DS2-4 (UDP) | RF / XGBoost / LightGBM | 100% |

**关键发现：**
- RF 和 XGBoost 在大多数数据集上表现出最高的准确率、精确率、召回率和 AUC-ROC
- KNN 执行时间最短（因为无训练阶段）
- DS1-6（慢速 DoS 攻击）对所有分类器最具挑战（RF 准确率仅 90.78%）
- 训练时间差异巨大：RF 最高达 738s（DS1-6），KNN 仅 7.6s（DS1-3）

### 6. 讨论

#### 坦诚的局限性分析

| 局限性 | 具体问题 | 改进方向 |
|--------|---------|---------|
| **二分类过于简化** | 仅区分"攻击"和"正常"，不区分攻击严重程度 | 多分类模型 + 置信度评分 |
| **H=0 硬性判断** | 当所有检测模型标记为攻击时认定恶意，可能误报 | 加入上下文参数（异常持续性、历史行为、操作上下文） |
| **Max 更新偏差** | 仅当 NDA > Max 时更新，对临时峰值/误报敏感 | 动态阈值机制 + 历史检测模式 + 模型置信度 |
| **数据集质量依赖** | 数据不平衡导致高误报或漏报 | 数据增强、对抗训练、持续学习 |
| **计算开销** | ML-IDS 导致 CPU/RAM/时间开销显著增加 | 高性能计算基础设施、模型精简 |

#### 关于计算开销的重要说明

作者承认：
> "提出的模型通常处理更少的授权请求（在存在入侵时），并且始终需要比通用授权模型更高的计算资源，特别是在 CPU 和 RAM 使用方面。这种增加的开销源于机器学习入侵检测系统的集成，它需要额外的处理能力进行实时异常检测、特征提取和对潜在恶意请求的分类。"

### 7. 结论与未来工作

#### 核心结论

1. **ITM-RAH** 模型通过信任评估 + ML-IDS + EHO 优化，在授权灵活性上显著优于传统 RBAC/ABAC。
2. 安全性提升以计算资源开销为代价——这是核心 trade-off。
3. 实验结果受数据集特征、超参数选择、硬件性能等多因素影响。

#### 未来工作方向

1. **深度学习集成：**
   - **RNN（循环神经网络）**：分析序列化攻击模式
   - **Autoencoder + GAN（生成对抗网络）**：检测零日攻击
   - **强化学习（RL）**：动态自适应调整授权策略
2. 与更先进的授权框架（如论文[33]中的方法）对比
3. 大规模企业 Hadoop 环境中的实际部署验证
